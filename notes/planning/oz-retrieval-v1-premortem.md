## Launch-Blocking Tigers

| # | Risk | Likelihood | Impact | Mitigation | Owner | Deadline | Status |
|---|---|---|---|---|---|---|---|
| T-01 | **Golden query suite doesn't exist yet for retrieval assertions.** Success metrics in the PRD (top-K precision, code-entry-point surfacing, package precision, notes surfacing) all depend on a fixture suite that can assert membership on `context_blocks`, `code_entry_points`, and `implementing_packages`. The existing `TestRoutingAccuracy` harness asserts agent choice only. Without a retrieval suite we cannot verify any P0 acceptance criterion and cannot detect regressions when tuning trust boosts. | High | Critical | Ship the `04_retrieval` golden suite as part of Sprint 1 before any retrieval selection code is written. Suite needs at least 8 queries covering: code-implementation ("how is drift detection implemented?"), spec-intent ("what is the review gate for?"), docs-only ("where is the build process documented?"), notes-rationale ("why did we reject embeddings?"), ambiguous / low-relevance. Each case asserts top-K membership — not exact order — to avoid over-fitting the suite to defaults. | oz-coding | Before Sprint 2 | Open |
| T-02 | **Default trust boosts and `min_relevance` are guesses.** The ADR-4 values (`specs=1.3`, `notes=0.6`, `min_relevance=0.05`, `agent_affinity=1.2`) are reasoned but not calibrated. If the first golden-suite run shows poor precision, every sprint after 1 is chasing a moving target because the math and the tuning are landing together. This mirrors the ADR-0003 threshold story ("0.85 is arbitrary until calibrated"). | High | High | Sprint 2 locks defaults by running the golden suite under a small grid over `(min_relevance, trust_boost_notes, agent_affinity_boost)` and picking the configuration that maximises top-K precision without dropping recall on spec-only queries. ## Fast-Follow Tigers

| # | Risk | Likelihood | Impact | Planned Response | Owner | Status |
|---|---|---|---|---|---|---|
| T-03 | **Body-token loading tanks query latency.** `spec_section`, `doc`, and `note` bodies are not in `graph.json`. Every query scores N blocks, each requiring a file read (or at least a tokenized-body lookup). On the oz workspace (~60 candidate blocks), this is 60 file reads per query at worst. p95 latency target in the PRD is 2× baseline; uncached on-demand reads could blow through that. | Medium | High | Implement an in-process tokenized-body cache keyed on graph `content_hash` alongside the on-demand loader. Cache invalidates on graph rebuild. Measure before and after Sprint 2 close; if p95 > 2× baseline with cache warm, pre-bake tokenized bodies into an on-disk sidecar (`context/retrieval-bodies.json`) — sidecar is derivative of `graph.json`, regenerated by `oz context build`, not committed. | oz-coding | Sprint 2 |
| T-04 | **Agent affinity boost over-rewards the read-chain.** Every file the winning agent reads gets ×1.2. For agents with wide read-chains (oz-spec reads most of `specs/`), the boost floats nearly every spec to the top regardless of query relevance — the same firehose in ranked clothing. | Medium | Medium | Verify on the `03_large` fixture: score the 10 widest read-chains, check that query relevance still dominates affinity in the top-K. If affinity dominates, reduce the boost to 1.1 or gate it on a minimum BM25 score (only boost blocks with non-zero query match). Lock the choice in Sprint 2's tuning step. | oz-coding | Sprint 2 |

## Track Tigers

*Monitor post-Phase 1 — trigger conditions noted.*

**T-05 — MCP packet consumers break on new fields.**
`code_entry_points` and `implementing_packages` (now populated instead of omitted) are additive, but a consumer validating against a strict schema may reject unknown fields. Trigger: any external MCP client we document (Claude Code, Cursor) surfaces a parse error. Response: add a `packet_version` field (default `"1"`, bumped to `"2"` on this change) and document field additions in `docs/api.md`.

## Paper Tigers

**P-01 — "Notes will drown higher-trust material at trust_boost=0.6."**
Not a real risk if BM25 does its job. `trust_boost` is multiplicative on an already-scored block, so a note can only beat a spec section when the note has a meaningfully higher raw BM25 against the query. If the notes are actually more on-topic, surfacing them is the correct behaviour — that's the whole point of reversing the blanket exclusion.

**P-02 — "Refactoring `scorer.go` helpers risks breaking routing."**
`normTF`, `avgFieldLengths`, `computeDF` are pure functions with existing unit tests. Parameterising them over a generic field-doc shape is a mechanical change covered by the current test matrix. If the routing golden suites pass after the refactor (required by R-05), routing is not broken.

## Elephants

**E-01 — Three adjacent bugs listed in PRD §9 (parser-mangling-skills, review-state, rules-not-node-ized) affect retrieval quality even if we ship V1 without them.**
The parser bug produces garbled tokens that degrade BM25F for routing and (now) retrieval. The rules-not-node-ized gap means `rules/coding-guidelines.md` is invisible to retrieval — a file referenced by every agent that the caller can never retrieve. V1 can ship without these, but acceptance numbers on the golden suite will be worse than they should be. Called out so we don't read V1 results as final.

**E-02 — We have no mechanism to measure retrieval quality against human judgement.**
The golden suite asserts membership in top-K, which is necessary but not sufficient. A block can be in top-K and still not be what a developer would have picked first. This is a V2 concern (human eval harness, inter-rater agreement) but worth naming so we don't treat "suite green" as "retrieval good."
